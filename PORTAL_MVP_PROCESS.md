# Портал личных кабинетов и конструктор чат-ботов (MVP)

## 1. Цель проекта

Создать веб-портал с личными кабинетами, в котором пользователи могут:
- создавать и настраивать Telegram-ботов;
- визуально описывать бизнес-процессы бота в формате BPMN (блоки/стрелки);
- публиковать процесс в 1 клик;
- запускать сценарий в работу сразу после публикации.

## 2. Роли пользователей

- Админ
- Менеджер
- Агент поддержки

## 3. Авторизация

- Логин/пароль
- OAuth: Google, Telegram
- 2FA на этапе MVP не требуется

## 4. Монетизация

- Подписка
- Тарифы
- Лимиты по тарифу:
  - количество ботов;
  - количество сообщений;
  - количество интеграций

## 6. Функциональный объем MVP

- Личный кабинет пользователя
- Управление ролями в рамках аккаунта/организации
- Создание и настройка Telegram-бота
- BPMN-конструктор процессов (блоки/связи)
- Валидация процесса перед публикацией
- Публикация сценария в 1 клик
- Исполнение опубликованного сценария в runtime
- Журнал событий (действия пользователя, публикации, ошибки, runtime-события)
- Базовая аналитика

## 7. Архитектурные принципы

- Модульная архитектура (сервисный подход)
- Отдельный слой адаптеров каналов (начать с Telegram)
- Отдельный слой интеграций (коннекторы)
- Версионирование бизнес-процессов
- Публикация активной версии без остановки сервиса
- Закладка на горизонтальное масштабирование

## 8. Хранение данных

- На этапе MVP: локальная база данных на текущем сервере
- Обязательно использовать миграции схемы БД
- Структуру БД проектировать с учетом дальнейшей миграции на отдельный сервер/кластер

## 11. UX-потоки (MVP)

1. Регистрация/вход -> создание организации -> выбор тарифа (или trial)
2. Создание бота -> подключение Telegram -> статус "черновик"
3. Проектирование процесса в BPMN-редакторе -> валидация -> сохранение
4. Публикация в 1 клик -> создание версии -> активация runtime
5. Мониторинг через журнал событий

## 12. Roadmap реализации

### Этап 1. Основа платформы (1-2 недели)

- Авторизация и роли
- Организация/аккаунт
- Базовый каркас личного кабинета
- Базовая модель тарифов/лимитов

**Статус:** базовый объём этапа 1 реализован в репозитории (FastAPI + React/Vite, SQLite).

**Сделано**

| Тема | Реализация |
|------|------------|
| Авторизация и роли | JWT (`/api/v1/auth/login`, `/api/v1/auth/register`); роли `admin`, `manager`, `support_agent` в `organization_users`; `/api/v1/me` отдаёт пользователя и роль; приглашение пользователей и смена роли только для `admin` (`require_admin`). |
| Организация / аккаунт | Модели `Organization`, `OrganizationUser`; `GET /api/v1/org`; связь пользователь ↔ организация при регистрации и в сидере тестовых данных. |
| Каркас ЛК | Фронт: сайдбар, разделы Вход / Профиль / Организация / Пользователи, JWT в `localStorage`; вывод данных через действия и `<pre>` (без отдельных страниц и react-router). |
| Тарифы и лимиты | Модели `Plan`, `Subscription`, `UsageCounter`; `GET /api/v1/billing/plan`, `GET /api/v1/billing/limits`; сидер создаёт план `trial` и счётчики использования. |

**Замечания**

- **OAuth** (Google / Telegram): в API есть заглушки, не реализованы (при необходимости — отдельная задача).
- **Фронт и роли:** роль в интерфейсе не отображается; пункты меню не скрываются по роли — ограничения админских действий только на бэкенде.
- **Несколько организаций на одного пользователя:** в `get_current_org_member` используется первая связь `OrganizationUser` для пользователя; сценарий выбора/переключения орг не делался.
- **Лимиты тарифа:** хранение и отдача через API есть; **создание ботов** и **echo в Telegram** учитывают лимиты тарифа; остальные сценарии (интеграции, шаги процесса) — в следующих этапах.
- **ЛК:** минимальный UI (JSON в блоке результата), без полноценных форм настроек аккаунта; отдельная продакшен-сборка и деплой в рамки этапа 1 не входили.

### Этап 2. Telegram и core бота (1-2 недели)

- Создание бота
- Подключение Telegram webhook
- Отправка/прием сообщений

**Статус:** минимальная реализация (MVP) в репозитории: боты в БД, `setWebhook`, приём `message` и echo-ответ, лог сообщений.

**Сделано**

| Тема | Реализация |
|------|------------|
| Создание бота | `POST /api/v1/bots`, список/просмотр/удаление; учёт лимита ботов по тарифу (`bots_used`); роли `admin` и `manager` для создания/удаления и подключения токена. |
| Webhook | `POST /api/v1/webhooks/telegram/{webhook_secret}` (без JWT); после ввода токена BotFather — `setWebhook` на URL из `PUBLIC_BASE_URL`. |
| Сообщения | Входящие сохраняются в `bot_messages`; при не превышении лимита сообщений по тарифу — исходящий ответ `Echo: …`; счётчик `messages_used` увеличивается на каждое входящее. |

**Замечания**

- **`PUBLIC_BASE_URL`:** публичный базовый URL API (переменная окружения бэкенда, без `/` в конце). Telegram должен достучаться до вебхука; локально `127.0.0.1` не подходит — используйте **https**-туннель (например ngrok) и выставьте `PUBLIC_BASE_URL` на выданный адрес.
- **Токен бота** хранится в SQLite в открытом виде (MVP); для продакшена нужно шифрование или секрет-хранилище.
- **Echo** — заглушка сценария; полноценный сценарий по BPMN/runtime — этапы 3–4.
- Поддерживаются только текстовые сообщения `message.text`.

## 13. Критерии готовности MVP

- Пользователь может зарегистрироваться и войти в личный кабинет
- Пользователь с нужной ролью может создать Telegram-бота
- Процесс бота можно собрать визуально, сохранить и опубликовать
- После публикации бот исполняет актуальный процесс
- Действия и ошибки фиксируются в журнале
- Ограничения по тарифу корректно применяются
